Dominando el Control de Versiones

Una guía técnica para modernizar tu flujo de trabajo con Git.

Open Source Distributed v2.4+

01. ¿Por qué Git?

Git no es solo una herramienta de respaldo; es una máquina del tiempo para tu código. Permite experimentos paralelos y colaboración masiva sin riesgo de pérdida de datos.

Al ser distribuido, cada desarrollador tiene una copia completa del historial, lo que garantiza redundancia y velocidad.

Los 3 Estados

  • Modified Archivos con cambios locales.
  • Staged Listos para la foto (Snapshot).
  • Committed Guardados en la base de datos.

02. Comandos de Supervivencia

Iniciar
git init

Inicializa un repositorio en la carpeta actual.

Preparar
git add .

Añade todos los cambios al área de preparación.

Confirmar
git commit -m "feat: ..."

Crea un punto de control en la historia del proyecto.

03. Estrategia de Ramas

Las ramas permiten aislar el desarrollo de nuevas funciones.

git checkout -b feature/login

Crea una rama llamada 'feature/login' y cambia a ella inmediatamente.

04. Ignorar archivos

Crea un archivo .gitignore para evitar subir archivos innecesarios:

  • /node_modules
  • .env
  • /dist o /build

Resumen de Buenas Prácticas

Acción Recomendación
Frecuencia de Commits Pequeños y frecuentes (Atómicos).
Mensajes Usa el presente imperativo ("Add feature" en vez de "Added").
Ramas Nunca trabajes directamente sobre main o master.
